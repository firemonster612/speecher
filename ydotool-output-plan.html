Speecher Linux Output Method and ydotool Setup Plan

This plan captures the current design direction for improving Speecher's Linux text delivery flow, especially for terminals and AI coding CLIs where clipboard paste is often the wrong behavior.

Goals

  • Make Speecher's output settings clear: typing methods and clipboard methods must not be presented as the same thing.
  • Give users a direct output method selector with an Automatic option.
  • Support reliable terminal and coding CLI output through ydotool after explicit setup.
  • Use native Linux authorization prompts for privileged setup steps.
  • Keep Speecher itself unprivileged at runtime.
  • Provide a clear disable path for virtual keyboard support.

Current Problem

The existing settings flow shows output as wtype, wl-copy, Qt clipboard, then separately shows wtype availability. This is misleading because wl-copy and Qt clipboard do not type into the active field; they only put text on the clipboard.

The UI should stop presenting backend internals as a passive status list. Users should instead choose an output strategy, and Speecher should expose setup only when a backend needs explicit configuration.

Settings UI

Add a new top-level Output section in settings. This section should be separate from refinement output format, because delivery method and text formatting are different concerns.

Primary Row

Label Description Control
Method How Speecher delivers final text. Dropdown

Default Dropdown Options

  • Automatic
  • Type with wtype
  • Type with ydotool, visible but disabled until setup passes
  • Copy with wl-copy
  • Copy with Qt clipboard

ydotool should appear in the dropdown from the start so users can see that it exists, but it should be grayed out until setup passes. Hovering the disabled option should show a tooltip such as Set up ydotool first. If the UI toolkit does not support tooltips on disabled combo box items directly, show the same message in a small status/help line under the dropdown when the user highlights or tries to choose the disabled option.

Virtual Keyboard Row

Label Description Controls
Virtual keyboard Harder setup, but most reliable for terminals, coding CLIs, and Wayland apps. Set up, then Disable in Speecher and Remove setup when enabled

Once verified, enable Type with ydotool in the dropdown and include it in automatic mode. If setup is later disabled in Speecher, keep the option visible but disabled again unless the user re-enables it.

Automatic Output Order

Automatic mode should try the methods in the order that matches the current design discussion: try the methods most likely to fail first, then fall back to the more reliable clipboard paths.

  1. wtype
  2. ydotool, only when setup is verified and enabled
  3. wl-copy
  4. Qt clipboard

The reason for this order is that typing into the active field is the preferred outcome. Clipboard fallback is reliable as data transfer, but it is not equivalent to typing and may be wrong for terminal workflows.

Why ydotool

Strengths

  • Works below X11 and Wayland by using a virtual keyboard device.
  • Better fit for terminals and AI coding CLIs than clipboard-only delivery.
  • Avoids relying on app-specific paste shortcuts.

Tradeoffs

  • Requires ydotoold and access to /dev/uinput.
  • Needs privileged setup.
  • Keyboard layout and arbitrary Unicode behavior can be imperfect.
  • Virtual keyboard access has real security implications.

Speecher should treat ydotool as an opt-in reliability upgrade, not as a default visible option that fails for users who have not configured it.

Speecher should not bundle ydotool. The setup helper should install it through the system package manager so users receive normal distro updates, dependency handling, and package ownership.

ydotool Architecture

ydotool is the client. ydotoold is the daemon that owns the virtual input device and talks to /dev/uinput. Speecher should never run the full app as root.

Runtime Shape

Speecher app, unprivileged
  -> ydotool client
    -> user-level ydotoold socket
      -> /dev/uinput
        -> focused app receives keyboard input

Preferred Service Model

Use a Speecher-owned user-level service where possible, so Speecher has a predictable socket path and status model across distros. 

[Unit]
Description=Speecher virtual keyboard daemon

[Service]
Type=simple
ExecStart=/usr/bin/ydotoold --socket-path=%t/.ydotool_socket --socket-perm=0600
Restart=on-failure
RestartSec=1

[Install]
WantedBy=default.target

%t resolves to the user's runtime directory. This keeps the socket under the logged-in user's session.

Native Authorization

Privileged setup must use the system-native authentication popup. Speecher should not show its own password prompt and should not ask users to run terminal commands as the normal path.

On Linux, this means using Polkit through pkexec. The Speecher UI remains unprivileged, explains the changes, and launches a narrow setup helper only for privileged operations. 

Speecher UI
  -> probe current status as normal user
  -> explain required changes
  -> pkexec /usr/libexec/speecher/speecher-ydotool-setup --install --user "$USER"
  -> desktop-native auth popup appears
  -> helper performs privileged setup
  -> Speecher re-probes and verifies

Polkit Action

<action id="local.speecher.setup-ydotool">
  <description>Set up Speecher virtual keyboard support</description>
  <message>Authentication is required to set up virtual keyboard input for Speecher.</message>
  <defaults>
    <allow_any>no</allow_any>
    <allow_inactive>auth_admin</allow_inactive>
    <allow_active>auth_admin</allow_active>
  </defaults>
</action>

Privileged Helper Scope

The privileged helper must be small, auditable, and argument-driven. It must not execute arbitrary shell supplied by the UI.

  • Detect supported package manager.
  • Install ydotool and ydotoold where package names require both.
  • Run modprobe uinput.
  • Write /etc/modules-load.d/speecher-uinput.conf.
  • Create a dedicated speecher-uinput group where needed.
  • Add the target user to speecher-uinput.
  • Write /etc/udev/rules.d/70-speecher-uinput.rules.
  • Reload and trigger udev rules.
  • Install or enable the Speecher-owned user service support files where appropriate.

Target User Handling

Because pkexec sanitizes environment variables, Speecher should pass the target username explicitly. The helper must validate the username against the system user database before using it.

Uinput Permissions

The preferred permission model is a dedicated group instead of adding the user to a broad input-related group. 

modprobe uinput
printf 'uinput\n' > /etc/modules-load.d/speecher-uinput.conf
groupadd --system speecher-uinput
usermod -aG speecher-uinput "$TARGET_USER"
KERNEL=="uinput", SUBSYSTEM=="misc", OPTIONS+="static_node=uinput", GROUP="speecher-uinput", MODE="0660", TAG+="uaccess"

After group changes, the current login session may not immediately have the new group membership. Speecher should verify immediately, but it must be ready to show a Needs sign out state when the setup is installed but not active for the current session.

Setup Flow

  1. User opens settings and sees the Virtual keyboard row.
  2. User clicks Set up.
  3. Speecher runs non-privileged probes.
  4. Speecher shows the required changes in plain language.
  5. User confirms.
  6. Speecher launches the setup helper through pkexec.
  7. The desktop shows the native authorization popup.
  8. The helper performs only the required privileged changes.
  9. Speecher starts or restarts the user-level ydotoold service.
  10. Speecher verifies the daemon connection.
  11. Speecher verifies actual typing in a controlled test field.
  12. If verification passes, Speecher marks ydotool enabled, enables the existing dropdown option, and includes it in automatic output.

Probe and Status Model

Status Meaning Primary Action
Not installed ydotool or ydotoold is missing. Set up
Installed, daemon not running The binaries exist, but the daemon/socket is unavailable. Start service
Needs uinput permission The daemon cannot open or use /dev/uinput. Configure permissions
Needs sign out Group membership or session permissions changed, but the current session has not picked them up. Sign out and back in
Ready Speecher can reach ydotoold and type through the virtual keyboard. Use or disable
Ready, layout caveat Typing works, but keyboard layout may affect output. Use with warning

Verification

Connection Test

First verify that the client can reach the daemon without typing into arbitrary focused applications. 

ydotool debug speecher-probe

Controlled Typing Test

Real typing verification should happen only after Speecher focuses a controlled test field in its own UI. 

ydotool type -- "speecher test"

Runtime Delivery

For actual text delivery, Speecher should avoid shell quoting problems by writing text to stdin. 

ydotool type --file -

Disable and Removal

After setup, Speecher should expose two separate user-facing actions. The first only disables use of ydotool inside Speecher. The second removes the setup that Speecher added to the system.

Action Requires Auth Effect
Disable ydotool for Speecher No Stops Speecher from using ydotool, removes it from automatic output order, and grays out Type with ydotool in the method dropdown.
Remove ydotool setup Yes, through native Polkit auth Stops the Speecher-managed service, removes Speecher-installed udev/module-load files, removes Speecher-specific access where safe, and returns the setup state to not configured.

User Flow

  1. User opens the Output section after ydotool has been set up.
  2. User sees Type with ydotool enabled in the method dropdown and sees virtual keyboard status as Ready.
  3. User clicks Disable in Speecher to stop using it without changing the system.
  4. Speecher immediately removes ydotool from automatic output and grays out the dropdown option.
  5. User can click Set up or Enable again to re-enable Speecher's use of an already working setup.
  6. User clicks Remove setup to uninstall or remove the system integration Speecher added.
  7. Speecher explains that removal requires administrator permission and launches the helper through pkexec.
  8. The native desktop authorization popup appears.
  9. The helper removes only Speecher-managed service files, udev rules, module-load files, and access changes that are safe to remove.
  10. Speecher re-probes status and leaves Type with ydotool visible but disabled with the tooltip Set up ydotool first.

Removal should not blindly uninstall the distro ydotool package if it may have been installed before Speecher or may be used by another app. The helper should track what Speecher installed or ask explicitly before removing packages.

Implementation Tasks

  1. Add explicit persisted output method settings.
  2. Replace the misleading Output status rows with a dedicated Output section.
  3. Split clipboard delivery into explicit wl-copy and Qt clipboard methods.
  4. Add a ydotool delivery backend using ydotool type --file -.
  5. Add automatic delivery orchestration in the order wtype, verified ydotool, wl-copy, Qt clipboard.
  6. Add a YdotoolSetup probe/status component.
  7. Add a small privileged setup helper launched through pkexec.
  8. Ship the Polkit action file for native authentication prompts.
  9. Ship or generate the user-level ydotoold service definition.
  10. Add setup, verify, disable, and remove-system-integration UI flows.
  11. Add tests for settings validation, delivery selection, automatic fallback order, and setup status parsing.

Security Decisions

  • Speecher must never run the full app as root.
  • The setup helper must not run arbitrary commands supplied by the UI.
  • ydotool must be installed through the system package manager, not bundled inside Speecher.
  • The UI must clearly state that virtual keyboard support allows Speecher to simulate keyboard input.
  • ydotool should remain opt-in and visible but disabled in the method dropdown until setup verification passes.
  • Native Polkit authentication should be used for privileged setup and removal.
  • Runtime typing must happen as the logged-in user through the configured daemon socket.

Open Questions

  • Whether Speecher should install its own user service on every distro or reuse a distro-provided user service when it is already correct.
  • How aggressive removal should be when other apps may also rely on uinput access.
  • How much UI detail to show for keyboard layout caveats without making setup feel intimidating.
